# Plan2Agent 실사용 검증 체크리스트

작성 기준일: 2026-07-06

문서 홈: [Plan2Agent Docs](README.md) · 사용자 시작점: [Quickstart](quickstart.md)

이 문서는 신규 기능 확장을 멈추고 핵심 파이프라인(idea → Gate A-D → 감독형 task 실행)을 실제 프로젝트로 검증하는 기간에 사용하는 기록 양식이다. 목적은 "어느 레이어를 살리고 어느 레이어를 접을지"를 감이 아니라 기록으로 결정하는 것이다.

## 사용 방법

- 실제 프로젝트 2~3개를 처음부터 끝까지(아이디어 → Gate D → task 실행 완료) 통과시킨다.
- 프로젝트마다 이 문서의 "프로젝트 기록 양식" 사본을 하나 만들어 채운다. 권장 위치: `plans/dogfooding/<project_id>.md` (하네스 상태인 `.plan2agent/` 내부에는 두지 않는다).
- 기록은 세션이 끝날 때마다 바로 채운다. 나중에 몰아서 복기하면 생략/우회 항목이 사라진다.
- 측정값은 기억이 아니라 산출물에서 가져온다: `p2a.mjs info`, `status.md` change log, `runs/run-index.json`, `p2a.mjs tasks list`.

## 측정 기준 5가지

### 기준 1 — 소요 시간과 명령 실행 횟수

아이디어 입력부터 Gate D 통과까지, 그리고 첫 task 실행 완료까지 실제로 얼마나 걸리고 몇 번의 명령을 쳤는지 기록한다. 도구의 절차 비용(ceremony)을 측정한다.

| 항목 | 기록 방법 |
| --- | --- |
| 구간별 소요 시간 | Gate A/B/C/D 각 통과 시각을 `status.md` change log 날짜 기준으로 기록 |
| 명령 실행 횟수 | `node .plan2agent/scripts/p2a.mjs ...` 실행 횟수를 구간별로 센다 (셸 히스토리 활용) |
| 대기 지점 | 사용자 승인/답변을 기다리며 멈춘 지점과 사유 |

판정 신호: 같은 구간에서 명령 실행 횟수가 프로젝트를 거듭해도 줄지 않으면 해당 구간의 명령 계약이 과하다는 신호다.

### 기준 2 — 게이트 적중률

각 게이트가 실제로 문제를 잡아냈는지, 아니면 통과 도장만 찍었는지 기록한다. 게이트의 존재 가치를 측정한다.

| 게이트 | 실제로 잡은 문제 (건수와 내용) | 도장만 찍음 (Y/N) |
| --- | --- | --- |
| Gate A (intake 결정) | | |
| Gate B (spec 승인) | | |
| Gate C (task graph 검증) | | |
| Gate D (리뷰 blocker) | | |

"잡은 문제"의 기준: 게이트가 없었다면 잘못된 상태로 다음 단계에 넘어갔을 것이 명확한 경우만 센다. 문구 다듬기 수준의 수정은 세지 않는다.

판정 신호: 2~3개 프로젝트에서 한 번도 문제를 잡지 못한 게이트는 자동 검증(validator)으로 강등하거나 인접 게이트와 합칠 후보다.

### 기준 3 — 생략·우회 목록

귀찮아서 건너뛰었거나, 정식 절차 대신 손으로 우회한 단계를 전부 기록한다. 절차와 실제 사용의 괴리를 측정한다. 이 기준은 정직하게 쓸수록 가치가 있다 — 생략한 것 자체는 잘못이 아니고, 생략됐다는 사실이 데이터다.

| 생략/우회한 단계 | 정식 절차 | 실제로 한 것 | 이유 |
| --- | --- | --- | --- |
| 예: run 기록 | `p2a.mjs runs record` | 기록 안 함 | task 1개에 필드가 너무 많음 |

판정 신호: 모든 프로젝트에서 반복적으로 생략되는 단계는 삭제·단순화·자동화 후보 1순위다. 특히 `proposals`, `eval`, `memory`, `orchestrate` 명령을 한 번도 자발적으로 쓰지 않았다면 그대로 기록한다.

### 기준 4 — task 무수정 실행 비율

Gate C에서 나온 task graph의 task 중, 에이전트가 프롬프트·범위·수용 기준을 고치지 않고 그대로 실행해 완료한 비율을 기록한다. task graph 품질(하네스의 최종 산출물 품질)을 측정한다.

| 항목 | 값 |
| --- | --- |
| 전체 task 수 | |
| 무수정 완료 | |
| 수정 후 완료 (무엇을 고쳤는지) | |
| 분할/병합 필요했던 task | |
| 실행 불가 판정 task | |

판정 신호: 무수정 완료 비율이 낮으면 원인이 task 분해 단계(`p2a-task-breakdown`)인지 상류 spec 품질인지 "수정 내용" 열로 구분한다. 분할/병합이 반복되면 task 크기 기준이 스킬 지침과 어긋난 것이다.

### 기준 5 — 스타일 교정 횟수

task마다 사용자가 생성된 코드의 스타일(네이밍, 에러 처리 방식, 추상화 수준, 주석 등)을 교정한 횟수를 기록한다. 스타일 계약 루프(`.plan2agent/style.md`)의 실효성을 측정한다.

| 항목 | 기록 방법 |
| --- | --- |
| task별 교정 횟수 | 감독 세션에서 스타일 지적을 한 횟수를 task 단위로 센다 |
| 교정 내용 | 무엇을 어떻게 바꾸라고 했는지 (style.md 반영 후보) |
| style.md 반영 여부 | 교정이 proposal을 거쳐 계약 파일에 반영됐는지 (Y/N) |

판정 신호: 프로젝트가 진행될수록 task당 교정 횟수가 줄어야 한다. 줄지 않으면 구현자가 style.md를 실제로 따르지 않거나 계약 표현이 모호한 것이다. 버그 지적은 세지 않는다 — 동작은 맞는데 "이렇게 말고 저렇게 써"에 해당하는 것만 센다.

## 프로젝트 기록 양식

프로젝트마다 아래 골격을 복사해 채운다.

```markdown
# 도그푸딩 기록: <project_id>

- 기간: YYYY-MM-DD ~ YYYY-MM-DD
- 아이디어 한 문장:
- 사용 CLI: (Claude Code / Codex / Gemini)

## 기준 1 — 시간/명령 횟수
(구간별 표)

## 기준 2 — 게이트 적중
(게이트별 표)

## 기준 3 — 생략·우회
(목록 표)

## 기준 4 — task 무수정 실행
(집계 표)

## 기준 5 — 스타일 교정
(task별 교정 횟수와 내용)

## 자유 메모
- 가장 아팠던 마찰 1가지:
- 예상 밖으로 좋았던 것 1가지:

종합 판정

프로젝트 2~3개의 기록이 모이면 아래 질문에 답하고, 결과를 다음 개발 우선순위로 옮긴다.

1. 절차 비용: 명령 횟수가 가장 많았던 구간은 어디이고, 그 구간의 명령을 하나로 합칠 수 있는가.
2. 게이트 구조: 4개 게이트 중 실제로 가치를 낸 게이트는 몇 개인가. 못 낸 게이트는 강등/병합하는가.
3. 레이어 존폐: proposals / eval / memory / orchestrate 중 실사용에서 자발적으로 쓰인 것은 무엇인가. 쓰이지 않은 레이어는 동결을 유지하는가, 접는가.
4. 산출물 품질: task 무수정 실행 비율을 올리기 위해 고칠 지점이 task 분해인가, spec인가, intake 질문인가.
5. 스타일 학습: task당 스타일 교정 횟수가 프로젝트 후반으로 갈수록 줄었는가. 줄지 않았다면 style.md 소비 지점과 계약 표현 중 어디가 문제인가.

검증 기간 중 개발 원칙

- 새 스키마, 새 서브커맨드, 새 provider 미러, 메타 레이어(proposals/eval/memory) 확장은 하지 않는다.
- 도그푸딩에서 발견된 마찰 제거(명령 단순화, validator 오탐 수정, 문서 정정)는 허용한다.
- 예외를 두고 싶으면 그 기능이 위 5가지 기준 중 어느 값을 개선하는지 먼저 적는다. 적을 수 없으면 하지 않는다.
